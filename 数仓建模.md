## 命名规则

### 表名

采用 “前缀-中缀-后缀" 的形式，用小写英文单词（除了特别常用的一般不用缩写），为了可读，单词之间用下划线分割。

- 前缀：分区方式，枚举值三选一
  - acc：只有 latest 一个分区，按时间增量存放，latest 指向最新时间的路径（分区方式 latest 很像，但是数据本质内容有区别，一般 sql 逻辑中会依赖自身）
  - finished：按时间分区，按时间增量存放，每个分区指向对应的时间的路径
  - latest：只有 latest 一个分区，按时间全量存放，latest 指向最新时间的路径（如果路径是 datetime=latest，说明是个不更新的表）
- 中缀：数据的内容，格式为“主题域*实体 [*属性]"的
  - 主题域：枚举值，无法确定时务必咨询分析师
  - 实体：反映数据内容实体，例如 message、group
  - 属性：可为空，his 代表拉链表，detail 代表详情表，agg 代表聚合表
- 后缀：更新频率

### 字段名

- 以 \_id 结尾的都代表用户账号体系的 ID，类型是 BIGINT，有特例，比如
- 以 status 结尾的代表枚举值，类型是 INT，一般 1-based
- 以 ts 结尾的代表时间戳，类型是 TIMESTAMP
- parquet 不支持 date 类型，时间字段一律用途 TIMESTAMP，可以区分精确到日或秒
- 以 is 或 has 开头的，都是布尔类型，表示是否

## 数据更新

### 四种数据处理

1. 从多个 latest 表计算出 latest 表
2. 从一个按天更新的 finished 表计算出一个 finished 表
3. 从多个 latest 表计算出 acc 表并和自己的前一个分区做累计
4. 从多个 latest 表中根据时间戳取出当天的

其中 ods 分区 datetime

- latest/acc 1h 表，datetime 为 utc 时间
- finished 1d 表，datetime 为北京时间，更新时间为凌晨 2 点左右
- finished 1h 表，datetime 为 utc 时间（暂时没有用到）

### 执行步骤

1. 检查依赖表是否 ready → 调度器中的 sensor 节点
   - 如果依赖了结果表自身（只能是 acc 表且依赖分区是 T-1），检查 T-1 是否存在
   - 如果依赖表非结果表，
   - 如果依赖表是 acc 表或 latest 表
     - 如果依赖表是按天更新， 检查 latest 分区对应的 yyyyMMdd 是否等于 T
     - 如果依赖表是按小时更新，检查 latest 分区对应的 yyyyMMddHH（转成 CSTChina）是否大于等于 T+1 的 00 点
     - 如果依赖的分区是 T，另外检查 create_ts 字段是否存在
   - 如果依赖表是 finished 表
     - 如果依赖的分区是 T
     - 如果依赖表按天更新，检查 T
     - 如果依赖表按小时更新（非 private_udp_prod 表），检查 T 的 00 到 T 的 23（转成 CSTChina））的数据
     - 如果依赖的分区是 all，检查整表
2. 执行结果表的逻辑  → 调度器中的 logic 节点
   - 如果表不存在，或者表的 schema 变了，因为是外表，直接删表重建，不动历史数据文件，这里非 latest 表都会有风险
   - 如果是 acc 表且 latest 分区已指向 T，重跑 T，需要先把分区 latest 指向 T-1
   - 针对每个依赖表，根据依赖分区（finished 1h 表需把 00-23 转成 CSTChina）建立临时视图，然后替换原 sql 中的 db.table 部分
   - 运行 sql，写入 T
   - 新增或更新分区
3. 执行质量指标的逻辑  → 调度器中的 metric 节点
   - 检查指标是否 OK，是否需要 block pipelien → 调度器中的 quality 节点点

### 依赖分区选择

- 如果依赖表是 acc 表，依赖分区是 latest；当 acc 表依赖了自己，分区是 T-1；当从最新全量里按 create_ts 筛选出 execution_date 时，分区是 T
- 如果依赖表是 latest 表，依赖分区是 latest；当从最新全量里按 create_ts 筛选出 execution_date 时，分区是 T
- 如果依赖表是 finished 表，当 datetime=execution_date 时，依赖分区是 T；当依赖整表时，依赖分区是 all

  注意这里的 T，是 execution_date，2020-12-22 执行时，传入的 T 是 20201221。参考 [](https://airflow.apache.org/docs/apache-airflow/stable/concepts.html#execution-date)

## 血缘关系

以数仓里的“表“为血缘关系的分析对象，血缘关系存在查询语句和 job 的逻辑里，表的生成由“数据处理任务“完成，形式为 SparkSQL。

### 实现

- grammar => lexer+parser : generated by ANTLR
- query => tokens : by lexer
- tokens => ast (abstract syntax tree) : by parser
- ast => dependency, lineage graph, etc : by post processing

## 质量监控

通过计算并观察一张表的一些统计指标来实现模型质量监控，指标的值可以和设定的阈值进行比较，判断质量合格与否。

### 指标定义

1. 筛选条件（可选，可多个）

   - 定义了指标计算的范围，通过筛选条件，可以从表中选取特定行进行观察。如果未定义筛选条件，则观察全表

2. 聚合逻辑（必选，一个）

   - 定义了一个聚合函数（例如 sum，count，count_distinct）
   - 定义了作用的列（例如 city_name）

3. 评估逻辑（可选，一个）

   - 定义了期望，可能场景包括
     - 指标值与数字进行值比较；
     - 指标值与 T-1 的同个指标的值进行比较
     - 指标值与历史同个指标的平均（MA7，MA15，MA30）进行比较
     - 比较结果经过 a set of logical AND and logical OR
   - 期望的比较支持
     - lower bound : 大于，大于等于
     - upper bound: 小于，小于等于
     - equality: 等于，不等于
     - range: 正负范围，例如 T-1 的+-10，这里语法上写作 metric > T-1 - 10 AND metric < T-1 + 10
     - percentage, stability: T-1 或 MA 的百分比，例如 MA7 \* 5% ，这里语法上写作 MA7 \* 0.05
   - 定义了效果，表示不满足期望时是否会 block 下游的任务
     - 筛选条件和聚合逻辑会转化成 sparksql 语句执行。

如果无评估逻辑则直接观察。如果有评估逻辑，则进行判断符合不符合期望。

### 期望表达式

- metric > 2 表示，通过筛选和聚合计算后的指标值，期望大于 2
- metric < 10 OR metric < MA7+10 表示，通过筛选和聚合计算后的指标值，期望小于 10 或者小于 7 日平均+10
